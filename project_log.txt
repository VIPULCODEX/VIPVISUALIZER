WHAT WE DISCUSSED ABOUT NOVELTY
--------------------------------------------------------------------------------

Question:
  - Is the current algorithm new or already existing?

Answer:
  - The exact name/form "Mixed Induced-Acyclic Matching (MIAM) for kidney exchange"
    does not appear to be a standard established algorithm.
  - However, most ingredients already exist:
      * kidney exchange cycle/chain packing,
      * induced matching,
      * acyclic matching,
      * greedy algorithms,
      * FPT branching,
      * kernelization,
      * ILP-based kidney exchange optimization.
  - Therefore, we should not claim that induced matching, acyclic matching, or kidney
    exchange optimization itself is new.

Safe novelty claim:
  - The project proposes a conflict-aware hybrid kidney exchange model inspired by
    induced matching, acyclic matching, and FPT kernelization.
  - The novelty should be framed as the combined model + implementation pipeline,
    not as inventing the underlying graph problems.


--------------------------------------------------------------------------------
IMPORTANT WEAKNESS FOUND IN CURRENT MIAM
--------------------------------------------------------------------------------

Current MIAM implementation weakness:
  - It mostly selects patient-pair vertices.
  - Real kidney exchange optimization selects exchange cycles/chains.
  - Therefore, current MIAM is a prototype abstraction, not yet a full realistic
    kidney exchange optimization algorithm.

Conclusion:
  - The direction is not fake or useless.
  - But to make it defensible, the algorithm should move from selecting individual
    patient pairs to selecting disjoint exchange cycles/chains.

  - Use in project:

--------------------------------------------------------------------------------
PROPOSED NEW HYBRID ALGORITHM
--------------------------------------------------------------------------------

Recommended name:
  - PS-KCP: Preference-Stable Kernelized Cycle Packing

Purpose:
  - Solve a realistic kidney exchange problem faster in practice by reducing the
    candidate cycle space before exact search.

Core idea:
  - Do not select individual patient-pair vertices.
  - Generate valid exchange cycles first.
  - Treat each cycle as one candidate exchange.
  - Build a conflict graph over candidate cycles.
  - Use stability/preference pruning + kernelization + FPT branching.

Algorithm steps:
  1. Build compatibility graph G.
  2. Enumerate all valid 2-cycles and 3-cycles.
  3. Convert each cycle into a candidate exchange/meta-node.
  4. Compute candidate cycle weight:
        weight = transplant count
               + survival benefit
               + diagnosis urgency
               + organ health score
               + stability/preference bonus.
  5. Build cycle-conflict graph H:
        two cycle candidates conflict if:
          - they share a patient pair,
          - they share simulated hospital/crossmatch resources,
          - selecting both violates induced/conflict constraints,
          - one is preference-dominated by another available cycle.
  6. Apply preprocessing/kernel rules:
        - remove dominated cycles,
        - keep top candidates per patient,
        - remove unstable low-value cycles,
        - force include isolated high-value cycles when safe.
  7. Run greedy selection to get a fast lower bound.
  8. Run FPT/branch-and-bound on the reduced kernel.
  9. Run local improvement/repair:
        - try replacing one chosen cycle with two better non-conflicting cycles,
        - reduce stability violations if possible.
 10. Return selected cycles, total transplants, total weight, stability violations,
     kernel size, runtime, and comparison against greedy/current MIAM.


--------------------------------------------------------------------------------
WHY PS-KCP IS A BETTER PROJECT DIRECTION
--------------------------------------------------------------------------------

Why it solves a real problem:
  - Kidney exchange does not simply choose patients; it chooses executable cycles/chains.
  - Real exchanges may fail or be unstable because patients prefer other donors or
    because cycles conflict over patients/resources.
  - PS-KCP directly selects disjoint exchange cycles under preference and conflict
    constraints.

Why it is not just novelty for novelty's sake:
  - It responds to an actual gap between:
      * theoretical matching constraints,
      * practical kidney exchange cycles,
      * modern stability-based KE-Opt research,
      * deployment/demo constraints on small to medium datasets.

What not to claim:
  - Do not claim a universally faster algorithm for general KE-Opt.
  - Do not claim polynomial-time solution for NP-hard stable kidney exchange.
  - Do not claim induced matching or kidney exchange optimization is newly invented.

Safe performance claim:
  - "PS-KCP improves practical runtime on small and medium bounded-cycle kidney
    exchange instances by reducing the candidate cycle graph before exact search."

Complexity framing:
  - 2/3-cycle generation: O(n^3) worst case.
  - Preprocessing/kernelization: polynomial.
  - Exact search: O(2^r), where r is the reduced kernel size, not the original
    number of candidate cycles.
  - Practical improvement comes from making r much smaller than the full candidate
    cycle count.
